RESTful API 设计

产品经理需要理解API的基本结构，才能正确设计数据交互需求。以下是一个用户信息的API接口示例：

// GET /api/users/123
// 获取用户详情

{
  "code": 200,
  "message": "success",
  "data": {
    "id": 123,
    "username": "zhangsan",
    "email": "user@example.com",
    "avatar": "https://cdn.example.com/avatar.jpg",
    "created_at": "2024-01-15T08:30:00Z",
    "profile": {
      "nickname": "张三",
      "bio": "产品经理 @TechCorp",
      "location": "北京"
    }
  }
}
💡 PM 必读

code 字段：业务状态码，200表示成功。注意这不是HTTP状态码。

data 结构：定义了返回的数据格式，产品经理需要在PRD中明确字段含义和是否必填。

SQL 数据查询

学会基础SQL，可以自己查询数据验证假设，更好地理解数据结构：

-- 查询近7天注册用户数量（按天分组）
SELECT 
    DATE(created_at) as date,
    COUNT(*) as user_count
FROM users
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY DATE(created_at)
ORDER BY date DESC;

-- 查询结果示例：
-- | date       | user_count |
-- |------------|------------|
-- | 2024-01-20 | 156        |
-- | 2024-01-19 | 203        |
-- | 2024-01-18 | 189        |
💡 PM 必读

WHERE 条件：筛选条件决定数据范围，产品经理提数据需求时需要明确时间范围。

GROUP BY：分组统计常用于分析报表，如日活、留存等。

React 组件示例

理解组件化思维，有助于更好地拆解产品需求和设计原型：

// 用户卡片组件
function UserCard({ user, onFollow }) {
  return (
    <div className="user-card">
      <img 
        src={user.avatar} 
        alt={user.name}
        className="avatar"
      />
      <div className="info">
        <h3>{user.name}</h3>
        <p>{user.bio}</p>
      </div>
      <button 
        onClick={() => onFollow(user.id)}
        className="follow-btn"
      >
        关注
      </button>
    </div>
  );
}
💡 PM 必读

props（属性）：user 和 onFollow 是传入组件的数据和回调函数，产品经理需要明确组件需要哪些数据支持。

事件处理：onClick 表示点击事件，交互设计时需要明确触发条件和反馈。

电商购物车重构

性能优化与用户体验提升

背景与挑战

某电商平台购物车页面加载缓慢，用户在添加商品后需要等待3-5秒才能看到更新。技术团队反馈是接口响应慢，但产品团队希望找到更快见效的方案。

技术决策过程

  • 问题诊断：通过浏览器开发者工具分析，发现主要是商品价格计算接口（涉及库存查询、促销活动计算）响应慢
  • 方案讨论：技术团队提出缓存方案，产品团队建议采用"乐观更新"策略
  • 最终决定：前端先展示预估价格，后台异步计算真实价格，价格差异在结算时校正
💡 关键学习点

产品经理不需要懂具体实现，但要理解"乐观更新"和"最终一致性"的概念，在用户体验和技术复杂度之间找到平衡点。

成果与反思

页面加载时间从3.5秒降至0.8秒，转化率提升12%。这个案例说明技术方案不一定非要"完美"，而是要在业务目标和资源限制之间做权衡。

社交App消息系统

实时通讯技术选型

背景与挑战

需要在App中增加私信功能，要求消息实时送达，同时支持历史消息漫游。技术团队给出了长轮询和WebSocket两种方案。

技术决策过程

  • 需求梳理：区分"必须实时"（聊天消息）和"可延迟"（系统通知）两类场景
  • 方案对比：长轮询实现简单但延迟高；WebSocket实时性好但需要维护连接
  • 资源评估：考虑服务器成本和团队技术储备，决定核心聊天用WebSocket，通知用长轮询
💡 关键学习点

学会区分不同场景的技术要求，不要把所有需求都堆到"实时"上。理解WebSocket和长轮询的基本区别，有助于在技术评审时提出有价值的问题。
