互联网产品是如何工作的

当你在手机App上点了一个按钮，到看到结果展示，这背后发生了什么？理解这个过程，是产品经理与开发团队高效沟通的基础。

🌐 一个请求的全旅程

想象你要给朋友寄一封信。你需要知道对方的地址（URL），把信投进邮筒（发送请求），邮局处理（服务器处理），最后送到朋友手中（返回响应）。互联网产品的数据交互，本质上就是这样的一套"请求-响应"机制。

用户访问网页的完整流程

输入网址或点击按钮
→
🌐 浏览器/App
组装请求信息
→
🖥️ 服务器
处理业务逻辑
→
🗄️ 数据库
查询/存储数据
→
📱 返回结果
展示给用户

💡 产品经理必知

这个流程中，任何一个环节出问题，都会导致用户体验受损：

  • 网络不好 → 请求发不出去（显示加载中...）
  • 服务器繁忙 → 处理变慢（响应时间延长）
  • 数据库慢 → 查询耗时（页面白屏）

⚡ 客户端与服务端的分工

互联网产品通常采用"客户端-服务端"架构。理解两者的分工，能让你更清楚地向开发团队表达需求。

客户端（前端）做什么？

客户端是用户直接接触的部分，包括网页浏览器、手机App、小程序等。它的核心职责是呈现界面和处理交互。

1

界面渲染

把服务器返回的数据转换成可视化的页面，包括文字、图片、按钮等。

2

交互响应

捕获用户的点击、滑动、输入等操作，并给出即时反馈。

3

数据请求

向服务器发送请求，获取需要展示的数据或提交用户操作。

4

本地存储

临时保存一些数据，如登录状态、用户偏好设置等。

服务端（后端）做什么？

服务端通常运行在远程的计算机集群上，用户看不到但它一直在工作。它的核心职责是处理业务逻辑和管理数据。

1

业务处理

验证用户权限、计算价格、生成订单等核心业务逻辑。

2

数据管理

与数据库交互，完成数据的增删改查操作。

3

接口提供

为客户端提供标准化的数据接口（API）。

4

安全保障

验证身份、防止攻击、保护敏感数据。

📡 HTTP：互联网的通用语言

客户端和服务端之间需要一种"语言"来沟通，这就是HTTP（超文本传输协议）。它规定了请求和响应的格式，就像信件的格式一样。

一个HTTP请求包含什么？

GET /api/products/123 HTTP/1.1
Host: www.example.com
Authorization: Bearer xxx
Content-Type: application/json

{ "quantity": 2 }
一个典型的HTTP请求示例

🔍 逐行解读

  • GET：请求方法，表示"获取数据"。还有POST（提交）、PUT（更新）、DELETE（删除）等
  • /api/products/123：请求路径，表示要访问的资源
  • Host：目标服务器地址
  • Authorization：身份验证信息，证明你是谁
  • { "quantity": 2 }：请求体，携带的具体数据

服务器如何回应？

HTTP/1.1 200 OK
Content-Type: application/json

{
  "code": 200,
  "data": {
    "id": 123,
    "name": "无线耳机",
    "price": 299
  },
  "message": "success"
}
一个典型的HTTP响应示例

💡 产品经理必知的状态码

  • 200：成功，请求正常处理
  • 400：请求参数有误，前端需要检查
  • 401：未登录或登录过期
  • 403：没有权限执行此操作
  • 404：请求的资源不存在
  • 500：服务器内部错误

🛒 实战案例：电商下单流程

让我们通过一个完整的电商下单流程，把前面的知识点串联起来。

点击"立即购买"后发生了什么？
1️⃣
客户端收集信息
商品ID: 123 | 数量: 2 | 用户Token
2️⃣
发送POST请求
POST /api/orders | 携带商品和地址信息
3️⃣
服务端处理
验证库存 → 计算价格 → 生成订单号
4️⃣
数据库保存
订单记录 | 扣减库存 | 待支付状态
5️⃣
返回结果，跳转支付
订单ID: 20240115001 | 应付金额: ¥598

⚠️ 异常场景思考

作为产品经理，你需要考虑这些边界情况：

  • 用户点击两次"购买"按钮，会生成两个订单吗？（需要防重复提交）
  • 提交时库存刚好售罄，如何提示用户？（并发库存控制）
  • 网络中断后重连，用户怎么知道订单是否成功？（订单查询机制）

🤔 本课思考题

请思考

你正在设计一个"文章点赞"功能。用户点击点赞按钮后，从客户端点击到页面上数字+1，这个过程涉及哪些步骤？哪些在客户端完成，哪些需要服务端参与？

A. 只需要客户端完成，把点赞数+1显示出来，不需要告诉服务器
B. 先告诉服务器，等服务端返回成功后，客户端再把数字+1
C. 客户端先+1（让用户感觉快），同时告诉服务器，即使服务器失败也不回退
D. 客户端先+1，同时告诉服务器，如果服务器返回失败则回退显示

✅ 正确答案是 D

这就是"乐观更新"策略：客户端先展示结果给用户（优化体验），同时异步通知服务端。如果服务端处理失败（如网络问题），客户端需要回退状态或提示用户重试。这种设计兼顾了用户体验和数据一致性。
