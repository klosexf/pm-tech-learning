数据一致性策略

在分布式系统中，数据一致性是一个核心挑战。本课程将深入讲解 CAP 定理、BASE 理论，以及强一致性和最终一致性的实现方案，帮助你理解如何在业务场景中做出正确的一致性选择。

❓ 为什么需要数据一致性

在单体架构中，所有数据操作都在同一个数据库中进行，通过数据库事务可以保证ACID特性（原子性、一致性、隔离性、持久性）。但在微服务架构中，每个服务拥有自己的数据库，这就带来了数据一致性的挑战。

💡 典型场景

电商下单场景：订单服务创建订单、库存服务扣减库存、支付服务处理支付。如果库存扣减失败但订单已创建，就会出现数据不一致。

数据不一致会带来严重后果：

  • 超卖：库存已售罄但用户还能下单
  • 重复支付：同一订单被多次扣款
  • 数据错乱：订单显示已支付但库存未扣减
  • 财务损失：退款成功但订单仍显示已支付

⚠️ 关键认知

分布式系统中，无法同时满足一致性、可用性和分区容错性（CAP 定理）。必须在一致性和可用性之间做权衡。

📐 CAP 定理与 BASE 理论

CAP 定理是分布式系统的理论基础，它指出在分布式系统中，一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）三者最多只能同时满足两个。

🎯

一致性 (Consistency)

所有节点在同一时刻看到的数据是一致的。写入操作完成后，所有读取都能获得最新值。

⚡

可用性 (Availability)

每个请求都能在合理时间内获得响应，不保证返回的是最新数据。

🌐

分区容错性 (Partition Tolerance)

系统在网络分区（节点间通信中断）时仍能继续运行。

🔀

权衡选择

实际系统通常选择 CP（一致性+分区容错）或 AP（可用性+分区容错）。

📊 CAP 组合示例

CP 系统：ZooKeeper、etcd、HBase（优先保证一致性）
AP 系统：Cassandra、DynamoDB、Eureka（优先保证可用性）
CA 系统：传统关系型数据库（单节点，不考虑分区）

BASE 理论是对 CAP 中 AP 方案的延伸，它提出：

  • Basically Available（基本可用）：系统出现故障时，允许损失部分可用性，核心功能仍可用
  • Soft state（软状态）：允许数据存在中间状态，不影响整体可用性
  • Eventually consistent（最终一致性）：不保证实时一致，但保证最终达到一致状态

💡 理解 BASE

BASE 是 "Basically Available, Soft state, Eventually consistent" 的缩写。可以理解为：在分布式系统中，接受暂时的数据不一致，优先保证系统可用，通过异步机制最终达到一致。

🔒 强一致性方案

强一致性要求任何时刻所有节点看到的数据都是一致的。在金融交易、库存管理等场景中，强一致性是必需的。

🎯 适用场景

银行账户转账、证券交易系统、核心库存扣减、订单支付状态。这些场景不允许任何数据不一致，必须保证实时一致性。

实现强一致性的主要方案：

📝

两阶段提交 (2PC)

协调者先询问所有参与者是否可以提交，得到全部确认后再执行提交。保证原子性，但性能较差。

📝

三阶段提交 (3PC)

在 2PC 基础上增加预提交阶段，减少阻塞时间，提高可用性，但实现更复杂。

✅ 优势

  • 强一致性保证，数据不会错乱
  • 理论成熟，实现方案完善
  • 适合金融等高可靠场景

⚠️ 挑战

  • 性能开销大，影响系统吞吐量
  • 协调者单点故障风险
  • 阻塞时间长，降低并发能力
  • 不适合跨异构系统

⏱️ 最终一致性方案

最终一致性允许数据在一段时间内不一致，但保证经过一段时间后，所有副本的数据最终达到一致。这是分布式系统中最常用的一致性模型。

🎯 适用场景

社交网站点赞数、电商商品浏览量、搜索引擎索引更新、缓存同步。这些场景可以容忍短暂的不一致，优先保证系统性能和可用性。

实现最终一致性的主要方案：

🔄

异步复制

主库写入后立即返回，异步将数据同步到从库。适合读多写少的场景。

📨

消息队列

通过消息队列异步通知其他服务更新数据，实现最终一致性。

🎯

TCC 模式

Try-Confirm-Cancel 三阶段，预留资源后确认或取消，性能较好。

🔄

最大努力通知

服务A完成后通知服务B，失败则重试，适合对一致性要求不高的场景。

✅ 优势

  • 性能高，不影响主业务流程
  • 系统可用性好，部分故障不影响整体
  • 扩展性强，适合大规模系统
  • 实现相对简单

⚠️ 挑战

  • 存在短暂的数据不一致窗口
  • 需要处理消息丢失和重复
  • 对账和补偿机制复杂
  • 用户体验可能受影响

🎭 Saga 模式详解

Saga 模式是一种长事务解决方案，将一个大的分布式事务拆分为多个本地事务，每个本地事务提交后立即释放资源，通过补偿操作处理失败情况。

💡 核心思想

Saga 将长事务拆分为多个短事务（本地事务），每个短事务立即提交。如果某个步骤失败，通过执行补偿操作（回滚）来撤销之前的操作，保证最终一致性。

Saga 有两种实现方式：

📋

编排式 (Choreography)

每个服务完成操作后发送事件，触发下一个服务执行。去中心化，松耦合。

🎮

编排式 (Orchestration)

由 Saga 协调器统一调度各服务执行。集中控制，易于管理。

如果 T3 失败，执行补偿：取消支付 → 恢复库存 → 取消订单

⚡ 补偿操作设计原则

  • 幂等性：补偿操作可以被安全地重复执行
  • 可逆性：每个操作都有对应的反向操作
  • 有序性：补偿按相反顺序执行（先执行的后补偿）
  • 可见性：补偿过程对用户可感知（如显示"订单处理中"）
对比维度 2PC/3PC Saga 模式
一致性级别 强一致性 最终一致性
性能 较低，阻塞式 较高，非阻塞
复杂度 协议复杂 业务逻辑复杂
回滚机制 自动回滚 需手动实现补偿
适用场景 短事务、强一致 长事务、最终一致

💼 产品经理决策指南

作为产品经理，你需要在需求定义阶段就明确数据一致性要求，这会直接影响技术选型和系统架构。

📋 决策检查清单

  • 业务影响：数据不一致会造成什么后果？是否可接受？
  • 用户感知：用户能否容忍短暂的不一致？
  • 修复成本：出现问题后人工修复的成本有多高？
  • 性能要求：业务对响应时间和吞吐量有什么要求？
  • 合规要求：是否有监管要求必须保证强一致性？

场景化决策建议：

💰

金融交易

必须强一致性，使用 2PC 或 TCC 模式

📦

电商库存

核心库存用强一致，普通商品可最终一致

👍

社交功能

点赞、评论用最终一致性即可

📊

数据统计

报表、统计最终一致性，允许延迟

🔔

消息通知

最大努力通知，允许丢失

🛒

订单履约

使用 Saga 模式，分阶段处理

💡 PRD 写作建议

在需求文档中明确标注：
1. 涉及的服务和数据
2. 一致性要求（强一致/最终一致）
3. 失败处理策略（重试/补偿/人工介入）
4. 用户体验设计（loading状态、错误提示）

与技术团队沟通要点：

  • 询问当前系统的一致性方案是什么
  • 了解失败场景的处理机制
  • 确认监控和告警是否完善
  • 讨论用户体验如何设计（如显示处理中状态）
  • 了解对账和补偿机制

📝 本章小结

数据一致性是分布式系统的核心挑战。本课程我们学习了：

  • CAP 定理：一致性、可用性、分区容错性不可兼得
  • BASE 理论：基本可用、软状态、最终一致性
  • 强一致性：2PC/3PC，适合金融等高可靠场景
  • 最终一致性：异步复制、消息队列、TCC，性能更好
  • Saga 模式：长事务拆分，通过补偿保证最终一致

🎯 核心原则

没有最好的方案，只有最适合的方案。根据业务特点选择一致性级别，在一致性、可用性、性能之间找到平衡点。
