数据表设计与 ER 图

学习如何设计规范的数据库表结构，掌握 ER 图（实体关系图）的绘制方法。通过电商订单系统的实战案例， 理解表与表之间的关联关系，为后续的数据分析和产品设计打下坚实基础。

📊 什么是 ER 图

ER 图（Entity-Relationship Diagram）是数据库设计的核心工具，用于可视化地表示数据实体及其之间的关系。 它是产品经理与开发团队沟通数据结构的重要桥梁。

💡 为什么产品经理需要学 ER 图？

  • 理解业务数据结构：清楚知道产品涉及哪些数据实体
  • 评估功能可行性：判断某个功能是否需要新增数据表
  • 优化产品方案：通过数据结构反推更合理的产品设计
  • 高效沟通：用技术语言与开发团队讨论需求

🔗 实体关系类型

在 ER 图中，实体之间存在三种基本关系类型，每种关系都有特定的符号表示：

1:1
一对一
一个实体对应另一个实体
如：用户与身份证
1:N
一对多
一个实体对应多个实体
如：用户与订单
M:N
多对多
多个实体互相映射
如：学生与课程

📋 数据表设计规范

1. 命名规范

-- 表名使用小写 + 下划线，复数形式
CREATE TABLE `users` (...);
CREATE TABLE `order_items` (...);

-- 字段名同样使用小写 + 下划线
user_id, created_at, is_deleted

2. 必备字段（每个表都应该有）

CREATE TABLE `example` (
    -- 主键，自增
    `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    
    -- 业务字段...
    
    -- 创建时间
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    -- 更新时间
    `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    -- 软删除标记（不用真正删除数据）
    `is_deleted` TINYINT DEFAULT 0,
    
    -- 主键：唯一标识每条记录，id字段值不能重复，也不能为空
    PRIMARY KEY (`id`)
) ENGINE=InnoDB;

3. 字段类型选择指南

📐 常用字段类型对照

  • INT/BIGINT：整数，如 ID、数量、状态码
  • DECIMAL(10,2)：精确小数，如金额（不用 FLOAT）
  • VARCHAR(n)：短文本，如用户名、手机号（n ≤ 255）
  • TEXT：长文本，如商品详情、评论内容
  • DATETIME/TIMESTAMP：日期时间，如创建时间
  • BOOLEAN/TINYINT：布尔值，如是否启用、是否删除

📝 实战案例：订单模块设计

以电商订单为例，展示完整的数据表设计过程：

Step 1：识别实体

订单模块涉及的主要实体：用户、商品、订单、订单项、收货地址

Step 2：确定关系

  • 一个用户有多个订单（1:N）
  • 一个订单包含多个商品（通过订单项关联，1:N）
  • 一个用户有多个收货地址（1:N）

Step 3：定义字段

-- 订单表
CREATE TABLE `orders` (
    `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    `order_no` VARCHAR(32) NOT NULL COMMENT '订单编号',
    `user_id` BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
    `total_amount` DECIMAL(10,2) NOT NULL COMMENT '订单总金额',
    `status` TINYINT DEFAULT 1 COMMENT '订单状态:1待付款 2待发货 3待收货 4已完成',
    `pay_time` DATETIME DEFAULT NULL COMMENT '支付时间',
    `created_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    -- 主键：唯一标识每条记录，id字段值不能重复，也不能为空
    PRIMARY KEY (`id`),
    -- 唯一索引：确保订单编号不重复，一个订单号只能对应一个订单
    UNIQUE KEY `uk_order_no` (`order_no`),
    -- 普通索引：加速按用户ID查询订单（如：查询某用户的所有订单）
    KEY `idx_user_id` (`user_id`),
    -- 联合索引：加速按状态和时间范围查询（如：查询待付款且最近创建的订单）
    KEY `idx_status_created` (`status`, `created_at`)
) ENGINE=InnoDB COMMENT='订单表';

🎯 产品经理必知要点

✅ 表设计检查清单

  • 每个表都有自增主键 ID 吗？
    主键是唯一标识每条记录的字段，通常使用自增的 BIGINT 类型，如 id = 1, 2, 3... 方便快速定位和关联查询。
  • 是否包含了 created_at 和 updated_at 字段？
    记录数据的创建时间和最后更新时间，便于追踪数据变更、排查问题、统计报表（如按时间段查询订单）。
  • 是否需要软删除（is_deleted）？
    用标记位代替物理删除，0 表示未删除，1 表示已删除。误删后可恢复，保留历史数据用于审计和数据分析。
  • 外键关联是否正确建立了索引？
    外键字段（如 user_id）经常用于关联查询，加索引可大幅提升查询速度，避免全表扫描导致性能问题。
  • 金额字段是否使用 DECIMAL 而非 FLOAT？
    DECIMAL(10,2) 存储精确小数，适合金额计算；FLOAT 有精度误差，可能导致 99.99 变成 99.989999，财务计算会出错。
  • 表和字段是否有清晰的注释说明？
    COMMENT 注释帮助团队成员理解字段含义，如 status 字段注释 '1-待付款 2-待发货 3-待收货 4-已完成'，避免歧义。

数据建模是产品设计的底层能力。掌握 ER 图和数据表设计， 能让你在产品方案讨论中更有底气，也能更好地预估功能开发的工作量。
