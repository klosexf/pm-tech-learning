数据字典与字段设计

数据字典是产品经理与开发沟通的重要工具。掌握字段类型、约束规则、命名规范，能够让你更准确地描述业务需求，减少开发返工。

📚 什么是数据字典

数据字典（Data Dictionary）是对数据库中所有数据元素的定义和描述的集合。对于产品经理来说，它是与开发团队沟通业务规则的「翻译手册」。

💡 为什么产品经理需要掌握数据字典

  • 精准沟通：用开发听得懂的语言描述需求
  • 避免歧义：字段定义清晰，减少开发返工
  • 验收依据：作为功能验收的参考标准
  • 文档沉淀：便于后期维护和交接

一份完整的数据字典通常包含以下内容：

🏷️

字段名称

英文字段名（如 user_id）和中文业务名称（用户ID）

🔤

数据类型

VARCHAR、INT、DATETIME 等字段类型定义

📏

长度/精度

字符串长度、数字精度、小数位等限制

⭐

约束规则

主键、外键、必填、唯一性等约束

📝

业务说明

字段的业务含义、取值范围、使用场景

📎

示例值

典型取值示例，便于理解字段用途

⭐ 字段约束规则详解

约束规则决定了字段的取值范围和业务规则，是数据质量的重要保障。

📋 约束类型说明

约束标识 约束名称 说明 示例场景
🔑 PK 主键 (Primary Key) 唯一标识每条记录，不能为空 user_id、order_id
🔗 FK 外键 (Foreign Key) 关联其他表的主键 订单表关联用户表
❗ NOT NULL 非空约束 字段必须有值 手机号、邮箱
🔷 UNIQUE 唯一约束 字段值不能重复 用户名、订单号
📇 IDX 索引 加速查询，可重复 手机号查询、时间范围
📋 DEFAULT 默认值 未指定时的默认值 status = 0
💡 产品经理提示

在需求文档中标注关键约束，特别是必填字段和唯一性要求，这些是开发最容易遗漏的地方。

字段详解：status 状态字段

📊

status 字段设计

字段名 status
数据类型 TINYINT (1字节整数)
默认值 1 (正常)
取值范围 0, 1, 2

业务含义：
0 禁用 - 用户被封禁，无法登录
1 正常 - 账号可正常使用
2 未激活 - 注册后未验证邮箱/手机

📝 字段命名规范

统一的命名规范能够大幅提升团队协作效率和代码可读性。

命名基本原则

  • 小写 + 下划线：使用 snake_case 格式，如 user_name 而非 userName 或 UserName
  • 英文为主：避免拼音，如用 status 而非 zhuangtai
  • 简洁明了：命名要有意义，如 mobile 优于 tel
  • 避免保留字：不要用 order、key 等数据库保留字

常用命名约定

场景 推荐命名 不推荐 说明
主键ID id user_id 表内统一用 id
外键ID user_id uid / userId 关联表名 + _id
创建时间 created_at create_time 统一用 _at 后缀
更新时间 updated_at update_time 统一用 _at 后缀
是否删除 is_deleted deleted / del_flag is_ 前缀 + 形容词
状态字段 status state / stat 统一用 status

✅ 命名检查清单

  • 所有字段名是否使用小写和下划线？
  • 时间字段是否统一使用 _at 后缀？
  • 布尔字段是否使用 is_ 前缀？
  • 外键是否使用 表名_id 格式？
  • 是否避免了数据库保留字？

🚀 产品经理实战指南

如何编写数据字典文档

在 PRD 中描述数据需求时，建议按以下结构组织：

-- 示例：订单模块数据字典（PRD 节选） 【订单表】t_order ├─ id: BIGINT PK - 订单ID，全局唯一 ├─ order_no: VARCHAR(32) UK - 订单编号，用户可见（如：202401151430001） ├─ user_id: BIGINT FK - 买家用户ID，关联 t_user.id ├─ total_amount: DECIMAL(10,2) - 订单总金额（元） ├─ status: TINYINT DEFAULT 0 - 订单状态：0待付款 1待发货 2已发货 3已完成 4已取消 ├─ pay_time: DATETIME - 支付时间，支付成功后更新 ├─ created_at: DATETIME - 下单时间 └─ updated_at: DATETIME - 最后更新时间 【订单商品表】t_order_item ├─ id: BIGINT PK ├─ order_id: BIGINT FK - 关联订单 ├─ product_id: BIGINT FK - 关联商品 ├─ product_name: VARCHAR(128) - 商品名称（快照，防修改） ├─ price: DECIMAL(10,2) - 下单时单价 ├─ quantity: INT - 购买数量 └─ subtotal: DECIMAL(10,2) - 小计金额（price * quantity）

与开发沟通的数据字段清单

1️⃣

确定字段类型

字符串长度是否足够？数值是否需要小数？时间是否需要时分秒？

2️⃣

明确约束规则

哪些字段必填？哪些是唯一的？是否需要索引加速查询？

3️⃣

定义状态值

状态字段有哪些取值？每个值的业务含义是什么？

4️⃣

确认默认值

新增记录时，各字段的默认值是什么？是否允许为空？

💡 字段类型选择技巧

  • VARCHAR：变长字符串，适合用户名、标题、描述等
  • INT/BIGINT：整数，适合数量、状态码、ID等
  • DECIMAL：精确小数，适合金额、比例等精度要求高的场景
  • DATETIME：日期时间，适合创建时间、更新时间等
  • TEXT：长文本，适合文章内容、备注等大量文字
  • TINYINT：小整数，适合状态、类型等有限取值

🎓 章节总结

📌 核心知识点

  • 数据字典是产品经理与开发沟通的「翻译手册」
  • 常用字段类型：字符串(VARCHAR)、整数(INT/BIGINT)、小数(DECIMAL)、时间(DATETIME)
  • 关键约束：主键(PK)、外键(FK)、非空(NOT NULL)、唯一(UNIQUE)、索引(IDX)
  • 命名规范：小写+下划线、表名前缀、时间用 _at 后缀、布尔用 is_ 前缀
  • PRD 中应包含核心表的字段定义，便于开发和验收
