API：数据的搬运工

API就像餐厅的服务员，负责在客户端（顾客）和服务端（厨房）之间传递信息。理解API的工作原理，是产品经理与开发团队沟通的核心技能。

🚚 API是什么？一个生活中的比喻

想象你走进一家餐厅点餐。你不会直接冲进厨房告诉厨师你想吃什么，而是通过服务员来传递你的需求。服务员接收你的订单（请求），送到厨房（服务端），然后把做好的菜（响应）端给你。

餐厅点餐 = API数据交互
👤
顾客
客户端/App
→ 点菜
🤵
服务员
API
→ 传单
👨‍🍳
厨师
服务端/服务器

💡 关键点

  • API是桥梁：连接客户端和服务端的"中间人"
  • 标准化沟通：API定义了请求的格式和响应的数据结构
  • 双向传递：既可以"点菜"（请求数据），也可以"买单"（提交数据）
  • 隐藏细节：顾客不需要知道厨房怎么做菜，只需要知道怎么点菜

🔄 API的完整工作流程

一个API调用通常包含四个步骤：构建请求、发送请求、处理请求、返回响应。让我们以"查询商品详情"为例，看看这个过程是如何进行的。

🛠️ RESTful API：四种基本操作

RESTful API是最常用的API设计规范，它使用HTTP协议定义了四种对资源的操作方式：增删改查。这就像对数据库里的数据进行操作一样。

GET 获取资源

从服务器获取数据，不会修改服务器上的任何内容。就像查菜单、看订单状态。

POST 创建资源

向服务器提交数据，创建新资源。就像下单、提交表单、发表评论。

PUT 更新资源

修改服务器上已有的资源。就像修改订单地址、更新个人信息。

DELETE 删除资源

删除服务器上的资源。就像取消订单、删除购物车商品。

💡 产品经理必知的URL设计规范

  • 资源名用复数：/users 而不是 /user
  • 全部小写：/products 而不是 /Products
  • 用斜杠表示层级：/users/123/orders 表示用户123的订单
  • 不用动词：用 GET /users 而不是 /getUsers

💼 实战场景：常见业务API设计

让我们看看真实产品中的API是如何设计的。理解这些，你就能更好地阅读接口文档，也能在需求评审时提出更专业的建议。

🛒
电商购物车场景

用户浏览商品、加入购物车、结算下单的完整流程涉及的API：

GET /products/{id} 获取商品详情
POST /cart/items 添加商品到购物车
GET /cart 获取购物车列表
PUT /cart/items/{id} 修改商品数量
DELETE /cart/items/{id} 删除购物车商品
POST /orders 创建订单
📋 核心接口详解

GET /products/{id} — 路径参数：id（商品ID）；响应：商品名称、价格、库存、图片、规格等

POST /cart/items — 请求体：productId、quantity、skuId（可选）；响应：购物车项ID、商品信息

PUT /cart/items/{id} — 路径参数：id（购物车项ID）；请求体：quantity（新数量）；响应：更新后的购物车项信息

DELETE /cart/items/{id} — 路径参数：id（购物车项ID）；响应：删除结果

👤
用户中心场景

用户注册、登录、管理个人信息的API设计：

POST /auth/register 用户注册
POST /auth/login 用户登录
GET /users/me 获取当前用户信息
PUT /users/me 更新用户信息
📋 核心接口详解

POST /auth/register — 请求体：phone/email、password、nickname（可选）；响应：用户ID、Token

POST /auth/login — 请求体：account、password；响应：用户信息、Token（用于后续请求身份验证）

GET /users/me — 请求头：Authorization（Bearer Token）；响应：用户昵称、头像、手机号（脱敏）、注册时间等

PUT /users/me — 请求头：Authorization；请求体：nickname、avatar（可选）；响应：更新后的用户信息

⚠️ 产品经理关注点

  • 接口粒度：是一个功能一个接口，还是多个功能合并？影响前后端工作量
  • 数据字段：接口返回哪些字段？哪些必填？需要PM在PRD中明确
  • 错误处理：网络失败、服务器错误时，前端如何提示用户？
  • 性能考虑：列表接口是否分页？一次返回多少条数据？

✏️ 实战练习：设计API接口

🎯

练习：为"文章点赞"功能设计API

假设你正在设计一个博客App，需要实现文章点赞功能。用户点击点赞按钮，文章的点赞数+1，再次点击取消点赞。请思考需要哪些API接口？

以下哪个API设计最合理？
A. 只需要一个 GET /articles/{id}/like 接口，每次调用点赞数+1
B. POST /articles/{id}/like 点赞，DELETE /articles/{id}/like 取消点赞
C. 只需要一个 POST /like 接口，通过参数控制是点赞还是取消
D. 用 PUT /articles/{id} 更新整个文章数据，包含点赞状态
✓ 正确答案是 B

点赞和取消点赞是两个独立的操作，分别对应创建（POST）和删除（DELETE）行为，符合RESTful设计原则。A选项无法实现取消点赞；C选项虽然可行，但不够清晰；D选项会导致每次点赞都要传输整篇文章数据，效率低下。

📝 拓展思考

除了点赞接口，实际场景中还需要考虑：

  • 如何判断用户是否已经点赞过？（需要在获取文章详情时返回 likeStatus 字段）
  • 点赞是否需要登录？（需要身份认证 Token）
  • 能否重复点赞？（后端需要做幂等性处理）
  • 点赞后如何实时更新点赞数？（需要考虑乐观更新或WebSocket推送）

💡 RESTful API 设计原则

  • GET - 获取资源，幂等操作，不改变服务器状态
  • POST - 创建新资源，非幂等，每次调用可能产生不同结果
  • PUT - 更新资源，幂等操作，整体替换
  • DELETE - 删除资源，幂等操作
