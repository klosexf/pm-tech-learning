数据库基础入门

理解数据库的基本概念，掌握关系型与非关系型数据库的区别。学习常用的 MySQL、Redis 等数据库的适用场景，为与开发团队沟通数据需求打下基础。

🗄️ 什么是数据库

数据库（Database）是存储和管理数据的仓库。想象一个电商 App，需要存储用户信息、商品信息、订单记录等海量数据，这些数据都需要有组织地保存在数据库中，才能方便地查询、修改和统计。

💡 产品经理理解要点

  • 数据持久化：用户关闭 App 后，购物车里的商品不会消失，这就是数据库的作用
  • 数据一致性：库存数量在数据库中只有一份，避免多用户同时购买时出现超卖
  • 数据查询：支持按条件快速查找，如"查找近30天北京用户的订单"

数据库的核心概念

📊

表（Table）

数据以表格形式存储，如用户表、订单表。每行是一条记录，每列是一个字段。

🔑

主键（Primary Key）

唯一标识每条记录的字段，如用户ID。就像每个人的身份证号，不可重复。

🔗

外键（Foreign Key）

建立表与表之间的关联。订单表中的 user_id 关联到用户表，表示"这个订单属于哪个用户"。

📑

索引（Index）

加速数据查询的数据结构。就像书的目录，让查找更快。常用在手机号、订单号等查询频繁的字段。

📂 常用数据库类型

不同类型的数据库适合不同的业务场景。作为产品经理，你需要理解它们的特点，才能参与技术选型讨论。

📊
关系型数据库 SQL

数据以表格形式存储，表与表之间可以建立关系。使用结构化查询语言（SQL）进行数据操作，适合数据结构固定、需要复杂查询的场景。

MySQL PostgreSQL Oracle
📂
非关系型数据库 NoSQL

数据以文档、键值对等形式存储，结构灵活。适合数据结构多变、需要高并发读写的场景，如用户行为日志、社交内容。

MongoDB Redis Elasticsearch

选型对比表

对比维度 关系型数据库 非关系型数据库
数据结构 固定的表结构 灵活的文档结构
扩展性 垂直扩展（增强单机） 水平扩展（加机器）
事务支持 完整支持 ACID 部分支持或最终一致
适用场景 订单、支付、用户核心数据 缓存、日志、社交内容

⚠️ 选型建议

  • 电商/金融类：主数据用关系型（订单、用户信息），热点数据用缓存（商品库存）
  • 内容/社交类：灵活的内容结构用文档型，搜索用 Elasticsearch
  • 实时类：排行榜、计数器用 Redis，消息队列用 Kafka

📝 SQL 基础入门

SQL（Structured Query Language）是操作关系型数据库的标准语言。作为产品经理，你不需要精通 SQL，但应该能读懂基本的查询语句，这样才能自主验证数据、排查问题。

最常用：SELECT 查询

SQL 查询用户订单列表
-- 从 orders 表中查询用户 U12345 的订单
SELECT
  order_id,
  total_amount,
  status,
  created_at
FROM orders
WHERE user_id = 'U12345'
  AND created_at >= '2024-01-01'
ORDER BY created_at DESC
LIMIT 10;

📌 SQL 关键字解析

  • SELECT：选择要查询的字段，* 表示全部字段
  • FROM：指定查询的数据表
  • WHERE：筛选条件，支持 AND、OR 组合多个条件
  • ORDER BY：排序，DESC 降序，ASC 升序
  • LIMIT：限制返回结果数量

数据统计：聚合函数

SQL 统计各状态订单数量和金额
SELECT
  status,
  COUNT(*) AS order_count,
  SUM(total_amount) AS total_revenue,
  AVG(total_amount) AS avg_amount
FROM orders
WHERE created_at >= '2024-01-01'
GROUP BY status;
🔢

COUNT(*)

统计记录数量。如：统计今日新增用户数

➕

SUM()

求和。如：计算今日总销售额

📊

AVG()

平均值。如：计算平均客单价

📈

MAX/MIN

最大/最小值。如：最高单笔订单金额

表关联：JOIN 查询

实际业务中，数据往往分散在多个表中。JOIN 用于关联查询多个表的数据。

SQL 查询用户及其订单信息
SELECT
  u.username,
  u.phone,
  o.order_id,
  o.total_amount
FROM users u
JOIN orders o ON u.user_id = o.user_id
WHERE o.status = 'paid'
ORDER BY o.total_amount DESC;

💡 JOIN 类型说明

  • INNER JOIN：只返回两个表匹配的数据（最常用）
  • LEFT JOIN：返回左表所有数据，右表不匹配填 NULL
  • RIGHT JOIN：返回右表所有数据，左表不匹配填 NULL

🔗 互动练习：数据库关系可视化

通过可视化方式理解数据库中表与表之间的关联关系。点击不同的表节点查看详情，观察外键如何建立关联。

📌 点击表节点查看详细信息
观察表之间的连线，实线表示外键关联，数字表示关系类型（1:1、1:N、M:N）。

💡 关系类型说明

  • 1:1（一对一）：如用户-用户详情，一个用户对应一条详情记录
  • 1:N（一对多）：如用户-订单，一个用户可以有多个订单
  • M:N（多对多）：如订单-商品，通过中间表（订单项）关联

💼 产品经理实战场景

掌握这些 SQL 知识，你可以自主验证产品逻辑、排查数据问题，不再完全依赖开发同学。

场景一：验证功能是否正常

SQL 检查优惠券是否被正确使用
-- 查看今天使用优惠券的订单
SELECT
  order_id,
  coupon_code,
  discount_amount,
  final_amount
FROM orders
WHERE coupon_code IS NOT NULL
  AND created_at >= CURDATE()
LIMIT 20;

场景二：分析业务数据

SQL 统计各渠道注册用户数量
SELECT
  channel,
  COUNT(*) AS user_count,
  COUNT(*) * 100.0 / (SELECT COUNT(*) FROM users) AS percentage
FROM users
WHERE created_at >= '2024-01-01'
GROUP BY channel
ORDER BY user_count DESC;

🎯 产品经理数据自查清单

  • 新功能上线后，检查数据是否按预期写入
  • 用户反馈问题时，快速查询相关订单/操作记录
  • 每日查看核心指标：新增用户、订单量、GMV
  • 验证 A/B 测试数据，确保分流比例正确
