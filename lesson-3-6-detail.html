阶段实战：设计电商系统架构

本课程将综合运用阶段三所学知识，从零开始设计一个完整的电商系统架构。涵盖系统架构设计、服务拆分、技术选型、数据库设计、缓存策略等核心内容，帮助你建立完整的架构设计思维。

🏗️ 系统架构概览

我们将设计一个支持日均百万PV、十万级并发的电商系统。采用微服务架构，前后端分离，通过API网关统一接入，各业务服务独立部署、独立扩展。

💡 设计目标

高可用（99.99%）、高性能（P99 < 200ms）、高扩展（水平扩展）、数据一致性（关键业务强一致，非关键最终一致）

架构设计原则：

  • 单一职责：每个服务只负责一个业务领域
  • 高内聚低耦合：服务间通过API通信，避免直接数据库访问
  • 水平扩展：所有服务支持多实例部署，应对流量高峰
  • 容错设计：服务降级、熔断、限流保障系统稳定性

🧩 服务拆分设计

根据电商业务特点，我们采用领域驱动设计（DDD）的思想进行服务拆分。以下是核心服务及其职责：

👤

用户服务（User Service）

用户注册、登录、个人信息管理、权限验证、用户等级体系

JWT认证 RBAC权限
📦

商品服务（Product Service）

商品信息管理、分类管理、品牌管理、商品搜索、价格计算

ES搜索 多级类目
📋

订单服务（Order Service）

订单创建、订单状态管理、订单查询、订单取消、订单统计

状态机 幂等设计
🏪

库存服务（Inventory Service）

库存扣减、库存预占、库存释放、库存查询、库存预警

Redis预扣 异步同步
💳

支付服务（Payment Service）

支付渠道管理、支付单创建、支付回调、退款处理、对账

多渠道 幂等控制
🎁

营销服务（Marketing Service）

优惠券管理、促销活动、积分系统、会员权益、推荐算法

规则引擎 实时计算

⚡ 服务间调用关系

  • 订单创建流程：订单服务 → 库存服务（扣减）→ 支付服务（创建支付单）
  • 商品详情：商品服务 + 营销服务（计算优惠后价格）
  • 订单支付回调：支付服务 → 订单服务（状态更新）→ 库存服务（确认扣减）

⚙️ 技术选型说明

技术选型需要综合考虑团队技术栈、社区活跃度、性能表现、运维成本等因素。以下是本系统的技术选型方案：

技术领域 选型方案 选型理由
开发语言 Java / Go Java生态成熟，Go适合高性能网关
微服务框架 Spring Cloud Alibaba 一站式解决方案，中文文档完善
API网关 Spring Cloud Gateway 性能优秀，支持动态路由
服务注册 Nacos 注册中心+配置中心一体化
数据库 MySQL 8.0 成熟稳定，支持读写分离
缓存 Redis 6.x Cluster 高性能，支持集群模式
搜索引擎 Elasticsearch 7.x 商品搜索、日志分析
消息队列 RocketMQ 阿里出品，高可靠，支持事务消息
容器编排 Kubernetes 容器化部署标准，弹性伸缩
监控告警 Prometheus + Grafana 云原生监控标准方案

🎯 选型原则

优先选择成熟稳定的技术，避免过于激进的方案；考虑团队技术储备，降低学习成本；关注社区活跃度，确保长期维护；预留扩展空间，支持业务快速发展。

🗄️ 数据库设计

每个微服务拥有独立的数据库，避免服务间直接访问对方数据库。以下是核心服务的数据库设计：

📋 用户服务数据库（user_db）

user 用户基础信息表
user_address 用户收货地址表
user_level 用户等级权益表

📦 商品服务数据库（product_db）

product 商品基础信息表
product_sku 商品SKU表
category 商品分类表
brand 品牌表

📋 订单服务数据库（order_db）

order_main 订单主表（按用户ID分表）
order_item 订单商品明细表
order_status_log 订单状态变更日志表

🏪 库存服务数据库（inventory_db）

inventory 库存主表（按商品ID分表）
inventory_log 库存变更流水表
inventory_preoccupied 库存预占表

💡 分库分表策略

  • 订单表：按用户ID取模分16张表，支持水平扩展
  • 库存表：按商品ID取模分8张表，热点商品分散压力
  • 历史订单：定期归档到历史库，主库只保留最近3个月

⚡ 缓存策略

缓存是提升系统性能的关键手段。我们采用多级缓存架构，从浏览器缓存到CDN、Nginx、Redis，层层加速。

🌐

浏览器缓存

静态资源缓存，减少重复请求

📡

CDN缓存

就近访问，加速静态资源

🚀

Nginx缓存

页面片段缓存，减轻后端压力

📦

本地缓存

Caffeine本地缓存，微秒级响应

🔴

Redis缓存

分布式缓存，数据共享

🗄️

数据库

持久化存储，最终数据源

核心缓存策略：

📦 商品详情缓存策略

商品信息变动频率低，适合长期缓存

  • Redis缓存：key=product:detail:{skuId}，过期时间24小时
  • 本地缓存：Caffeine缓存热点商品，过期时间10分钟
  • 缓存更新：商品编辑后发送MQ，异步更新缓存
  • 缓存穿透：空值缓存 + 布隆过滤器

🏪 库存缓存策略

库存数据敏感，需要保证一致性

  • Redis预扣：使用Lua脚本保证原子性
  • 库存预热：大促前将热点库存加载到Redis
  • 缓存同步：定时任务同步Redis与数据库库存
  • 库存回滚：订单取消后异步释放库存

👤 用户会话缓存

用户登录状态管理

  • Token存储：JWT + Redis黑名单
  • 用户权限：Hash结构存储权限列表
  • 购物车：Hash结构，field=skuId，value=数量
  • 过期策略：30分钟无操作自动过期

⚠️ 缓存设计原则

  • 缓存一致性：更新数据库后删除缓存，而非更新缓存
  • 缓存粒度：细粒度缓存，减少缓存失效范围
  • 降级策略：Redis故障时直接访问数据库，保证可用性
  • 热点处理：本地缓存应对热点Key，防止Redis单点压力

📝 实战总结

通过本实战项目，我们设计了一个完整的电商系统架构。以下是关键设计要点回顾：

🎯 架构设计要点

分层架构：接入层 → 应用服务层 → 数据层 → 基础设施层，各层职责清晰。微服务拆分遵循DDD原则，服务间通过API网关通信，数据隔离保证独立性。

核心设计决策：

  • 服务通信：同步REST用于查询，异步MQ用于写操作，Saga处理分布式事务
  • 数据一致性：关键业务（库存、支付）强一致，非关键业务最终一致
  • 高可用设计：多实例部署、熔断降级、限流保护、异地多活
  • 性能优化：多级缓存、数据库分库分表、异步处理、批量操作

💼 PM 架构思维

  • 需求分析：明确功能需求的同时，定义性能指标（QPS、响应时间）
  • 风险评估：识别单点故障、数据一致性风险，提前规划应对方案
  • 扩展预留：考虑业务增长，架构具备水平扩展能力
  • 成本意识：在性能和成本之间找到平衡点，避免过度设计
